Step 1 of 8 backend

Change graph engine: import/definition parsing and edge extraction

Build a deterministic, added-line-only dependency graph across changed files for JS/TS, Go, and Python.

graph.go parses each file's added lines only, extracting import specifiers per language and resolving them against a path/module index that handles relative imports, index/__init__ files, extension variants, and ambiguous-match rejection. It separately finds exported definitions and links importer files to a definition's file when the symbol name is unique repo-wide and at least four characters. order.go then consumes the resulting edges to topologically sort files within a cohort (StableOrder) and rank the top three earlier-cohort dependencies by edge weight (CohortDependencies), using Tarjan's algorithm to detect and safely bypass cycles.

⚠ Worth double-checking

  • Regex-based import/definition parsing can misresolve or miss cases outside the tested patterns; check whether false edges could silently reorder unrelated files.
  • Unique-symbol linking requires the symbol to appear in exactly one file's added lines and be four or more characters — verify this threshold doesn't produce spurious edges on common short identifiers.
  • Confirm cycle handling in StableOrder falls back to the original file order rather than producing a partial or crashing sort.

Step 3 of 8 backend

Wire change graph into the analysis pipeline

Apply reading-order and cohort-dependency graph output during both direct (ApplySeatbelts) and staged planning/assembly.

plan.go adds ReadingOrder and CohortDependencies to Settings, builds the graph once per staged plan when either is enabled, reorders each cohort's files with StableOrder, and — in AssembleStagedAnalysis — replaces the previously always-empty DependsOn with capped dependencies from CohortDependencies. validate.go extends ApplySeatbelts to accept the file list and a readingOrder flag so the same reordering applies to the direct (non-staged) path, reordering FileSummaries in lockstep with Files after existing dependency-remap logic runs. app.go simply forwards the two new config values into analyze.Settings.

⚠ Worth double-checking

  • Reordering can occur in two places (staged planning and the shared ApplySeatbelts safety pass) — confirm they don't conflict or double-apply for staged runs.
  • Verify FileSummaries stays correctly aligned with Files after reordering in both the direct and staged assembly paths.
  • Confirm DependsOn values assembled here always reference strictly earlier cohort indexes so downstream diagram rendering can't cite a later or self cohort.
